顺序访存单口Cache实现

访存设计

我们需要支持miss时的refill操作、CACHE指令写回操作、uncached的read/write操作，为优化访存的时序，我们考虑对写操作单独维护相关结构，使其与流水线并行处理，以得到加速的目的。

首先考虑只有uncached的read/write操作的情况。读写的过程必须是严格保序的。我们维护write queue来缓存uncached write相关信息，当需要执行uncached read时，必须等到write queue empty时才能发起读请求。

接下来我们加入CACHE指令写回操作。考虑CACHE指令存在的意义，即将原本缓存在cache中的数据写回到内存中，以提供给DMA等结构使用，保证其一致性。且CACHE指令之后对于该地址的读写行为属性可能由cached切换为uncached。因此必须严格保证CACHE指令和其之后的uncached read/write的顺序。所以我们控制所有的uncached read/write必须等待write buffer为空时才能发起。

之后我们考虑refill操作，本质上，refill操作是一种例外行为，其目的在于修复不可预知的cache line缺失行为。因此，我们只需要考虑其一致性相关行为。因为存在config0_k0切换导致的uncached段转化为cached段的行为，因此我们需要等待write queue empty时才发起读行为。而当write buffer不为empty状态时，可以推断得知当前指令与写回发生时指令之间没有发生uncached read/write行为，因此write buffer中的cache line是可以做前递的。为提升性能并且保证顺序，我们可以直接使用write buffer中的cache line重填回cache中。

最后我们单独考虑refill操作中的写回部分，它本质上是一种不可预知的例外行为，理论上它可以发生在任意时刻，因此我们只需要保证对该行的cached refill的重取顺序即可。

综上所述，我们给出相关依赖关系：

  1. uncached read依赖write queue empty和write buffer empty
  2. uncached write依赖write buffer empty
  3. cached read依赖write queue empty，需要考虑write buffer中的剩余cache line
  4. cached write无特殊依赖

queue设计

我们使用SpinalHDL自带的Mem存储器来实现queue。其中读使用readAsync异步读来实现。

我们规范queue的读写通过push和pop信号进行控制，他们都是Stream类型，通过valid/ready握手信号与Cache主体部分进行交互。

为了满足复杂需求的需要，我们还对queue的状态进行了相关的描述。我们记队首为front，队尾为end。

stage flag
empty front == end
will empty front + 1 == end
full end + 1 == front
will full end + 2 == front

uncached write设计

为了加速uncached write行为，我们使用了write queue。为了使得write queue实现与cached write一样的效果，我们给出如下的设计方式：

首先经过分析可知，uncached write在一个周期内完成，需要满足write buffer empty且write queue unfilled两个条件。为了优化时序，我们需要在stage 1进行判断，并对stage 2的行为给出决策。

针对write buffer empty，我们需要考虑是否会因为下一个阶段的写入导致write buffer的状态从empty转变为remaining。我们通过限制主状态机的行为来严格控制这一行为不会在流水线流动时发生。我们还需要考虑当前时刻stage 2是否即将使得write buffer empty。因为我们的write buffer是使用带tag版本的queue来实现，因此只需要按照queue判断will empty的方式进行即可。给出相应的表达式就是(empty || (willEmpty && pop.ready)) && !push.ready.

针对write queue unfilled，我们可以预知下一个阶段是否转变为write queue full状态。给出相应的表达式为(full || (willFull && push.ready)) && !pop.ready.

主控制状态机设计

我们给出如下状态：

  • stateBoot

  • cacheArbitrate

  • cacheInvalidate

  • cachedPreFetch

  • cachedFetch

  • cachedRefill

  • uncachedWait4Empty

  • uncachedRead

  • uncachedWaif4Unfilled

stateBoot

作为主要的仲裁与控制单元。

  • 没有访存请求：pass

  • CACHE指令：goto cacheAribitrate

  • cached读/写：

    • 如果hit，直接读/写，pass
    • 如果!hit，且此行!dirty，goto wait4Empty
    • 如果!hit，且此行dirty，goto wait4Empty

  • uncached写：

    • 如果write queue有空余，直接写入write queue，pass
    • 如果write queue满，goto wait4Unfilled

  • uncached读：goto wait4Empty

cacheArbitrate

本状态选择CACHE指令的调度策略。

  • 需要发生写回行为且dirty时，goto cachedPreFetch

  • 不需要发生写回行为时，goto cacheInvalidate

cacheInvalidate

本状态只有一个周期，目的为使得CACHE指令操作行无效，goto stateBoot
